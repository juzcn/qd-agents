动态上下文编排与动态工作流编排智能体设计方案
1. 总体设计目标
构建一个自适应智能体，能够根据用户需求动态决定是否使用工具、动态编排工具调用流程（包括顺序、并行、条件分支、循环），并在执行过程中根据中间结果和用户介入动态调整计划。核心设计原则：

渐进式披露：工具信息分层加载，避免上下文过载。

五步循环：标准化处理流程（判断→规划→代码生成→执行→回答）。

动态上下文编排：每个步骤按需构建上下文视图，区分用户可见历史与过程历史。

统一工具模型：支持 Function Tool、MCP、CLI、Skill 等多种形式。

用户协作：支持智能体主动澄清和用户主动介入，状态可恢复。

2. 总体架构
系统由以下核心模块组成：

模块	职责
工具注册表	存储所有工具的元数据（L0～L3），支持渐进式查询
上下文编排器	根据步骤需求，从会话历史（用户可见+过程）、工作记忆、工具注册表中提取并组装上下文视图
五步循环引擎	管理第一步到第五步的流程，调用大模型，处理交互与介入
工作记忆	存储当前循环的中间产物（工具列表、方案、代码、执行结果），支持版本和依赖追踪
执行沙盒	安全执行生成的 Python 代码，隔离环境，支持超时、资源限制
交互管理器	处理智能体主动提问和用户主动介入，维护状态暂停与恢复
3. 统一工具模型与渐进式披露
3.1 工具元数据分层
层级	内容	示例（天气工具）
L0	名称 + 一句话功能描述	get_weather: 查询指定城市的天气
L1	输入参数（名称、类型、必填）、输出格式、副作用类型	city: string，返回 {temp, condition}，只读
L2	运行时特征：预估耗时、并发限制、所需权限、错误码	耗时 0.5s，无并发限制，需网络权限
L3	详细文档、使用示例、边缘情况说明	示例调用、错误处理建议
3.2 不同工具形式的适配
工具形式	L0/L1 来源	调用方式（适配器）	特殊处理
Function Tool	函数签名 + docstring	直接 Python 调用	无
MCP	list_tools 返回的 name/description/inputSchema	MCP 客户端 call_tool	需维护连接
CLI	命令名 + --help 解析	subprocess 运行	沙盒限制路径、超时，白名单
Skill	SKILL.md 中提取名称、描述、输入上下文、输出说明	上下文注入（作为系统提示词）或展开为子工作流	内部可调用其他工具
3.3 渐进式披露流程
第一步：只加载所有工具的 L0（名称+描述）。

第二步：根据第一步输出的工具列表，加载这些工具的 L1 及必要 L2。

第三步：若需要生成代码，加载 L1（参数/返回值细节）；L3 仅在调试或重写时按需加载。

4. 历史分离：用户可见历史与过程历史
4.1 定义与存储
历史类型	内容	可见性	用途
用户可见历史	用户消息 + 智能体最终返回的自然语言回答	用户可见（对话界面）	保持对话连贯、指代消解
过程历史	五步循环内部记录：工具列表、方案、代码、执行输出、重写决策、错误栈等	仅系统/大模型可见	支持决策回溯、重试、调试
两类历史分别存储为序列，每条记录包含时间戳、所属步骤、依赖关系。

用户可见回答关联到产生它的过程历史 ID。

4.2 上下文编排中的使用原则
默认：只将用户可见历史（压缩后）作为对话上下文；过程历史仅按步骤需求选择性加载（例如第二步需要第一步的工具列表，第三步需要第二步的方案）。

用户主动介入：新消息追加到用户可见历史，过程历史中的相关条目被标记为“过时”，系统重新规划。

调试模式：仅当用户显式请求（如“/debug”）时，才将相关过程历史加载到上下文，生成解释性回答。

5. 五步循环详细设计
5.1 整体流程
text
用户输入
   ↓
[第一步] 判断是否需要工具
   ├─ 直接回答 → 输出给用户（结束）
   └─ 需要工具 → 输出工具列表
        ↓
[第二步] 根据工具列表加载L1，决定直接调用 or 生成自然语言方案
   ├─ 直接调用 → 生成结构化调用计划 → 执行（跳至第四步风格）
   └─ 自然语言方案 → 输出方案文本
        ↓
[第三步] 根据方案生成 Python 代码
   ↓
[第四步] 在沙盒中执行代码，捕获结果/错误
   ↓
[第五步] 根据执行结果生成自然语言回答
   ↓
返回用户
每个步骤之间设有检查点，可处理智能体主动提问和用户主动介入。

5.2 第一步：判断是否需要工具
输入：用户最新消息 + 最近2-3轮用户可见历史 + 所有工具的 L0（名称+描述）

大模型任务：判断自身知识能否回答。若能，直接输出最终回答；若不能，输出工具名称列表（去重）。

交互点：若问题模糊无法确定工具，可主动询问用户澄清。

输出：直接回答 或 工具列表（存入工作记忆）。

5.3 第二步：直接调用或生成自然语言方案
输入：第一步输出的工具列表 + 这些工具的 L1 详情 + 用户可见历史（完整但可压缩） + 工作记忆中的相关中间结果

大模型任务：

评估任务复杂度：若只需顺序调用1-3个工具且无分支/循环/并行，则选择直接调用，生成结构化调用计划（JSON 序列，指定工具和参数）。

否则，生成自然语言编排方案，描述步骤、依赖、条件、循环、异常处理。

交互点：若缺少必要参数或参数有歧义，主动询问用户；若涉及高风险操作（如删除文件），请求用户确认。

输出：调用计划 或 自然语言方案（存入工作记忆）。

5.4 第三步：生成 Python 代码
输入：第二步生成的方案 + 涉及工具的 L1（参数类型、返回值格式） + 用户可见历史中的格式约束

大模型任务：将方案转换为可执行的 Python 代码。代码必须：

导入工具适配器（tool_gateway）

实现方案中的顺序、并行（使用 concurrent.futures）、条件、循环

包含错误处理（try-except）

不包含危险操作（禁止 os.system, eval 等）

输出结果以打印或返回方式提供（如 print(json.dumps(result))）

输出：Python 代码文本（存入工作记忆）。

5.5 第四步：执行 Python 代码
输入：第三步生成的代码 + 工具网关 + 沙盒配置

执行器：

将代码写入临时文件，在沙盒子进程中运行（限制 CPU、内存、超时、网络白名单）。

捕获 stdout/stderr、返回值、退出码。

若执行中需要用户输入（如 input()），通过交互管理器暂停并请求用户输入。

若执行超时或触发安全策略，终止进程并记录错误。

输出：执行结果（状态、数据、错误日志）存入工作记忆。

5.6 第五步：生成自然语言回答
输入：第四步的执行结果 + 用户原始问题 + 最近几轮用户可见历史

大模型任务：将结构化结果转换为友好、简洁的自然语言回答。若执行失败，解释原因并提供可操作建议。不编造信息。

输出：最终回答，追加到用户可见历史。

6. 动态上下文编排机制
6.1 上下文视图结构（每步骤前构建）
层级	内容	来源	管理策略
核心层	当前步骤绝对必需的信息（如第一步的用户问题+L0）	用户可见历史、工作记忆、工具注册表	不可淘汰，步骤结束后可降级
支撑层	有帮助但非必需的信息（如旧对话摘要、工具L2特征）	用户可见历史（压缩）、过程历史	效用评分低于阈值则丢弃或压缩
压缩层	摘要或指针（如远距离会话摘要、工具L3文档ID）	用户可见历史压缩、过程历史指针	需要时通过指针展开
6.2 各步骤的上下文需求
步骤	核心层	支撑层	压缩层
第一步	用户最新消息+最近2轮用户可见历史 + 所有工具L0	更早会话摘要	无
第二步	第一步工具列表 + 那些工具的L1 + 用户可见历史（全）	工作记忆中相关中间结果	旧对话摘要
第三步	第二步方案 + 涉及工具的L1	代码生成模板（若有）	工具L3指针
第四步	代码本身 + 工具网关接口	工作记忆变量绑定	无（执行器不需大模型）
第五步	执行结果 + 用户原始问题 + 最近一轮用户可见历史	执行日志（可选）	之前步骤中间产物指针
6.3 上下文编排器工作流程
需求解析：根据步骤标识和分支，确定需要哪些信息类别及最大 token 预算。

检索：从用户可见历史（按相关性/时效性截取）、过程历史（按步骤依赖）、工具注册表（按层级）中提取原始信息。

压缩：对超过阈值的文本（如过长会话）调用轻量 LLM 生成摘要，保留关键实体和决策点；原始内容存入长期存储并返回指针。

组装：生成包含核心层、支撑层、压缩层的结构化上下文对象，注入特殊标记（如 <core>、<support>）以辅助大模型区分重要性。

缓存：将上下文视图暂存，供下一步骤参考（但下一步会重新构建，可能部分复用）。

6.4 用户主动介入时的上下文更新
用户新消息追加到用户可见历史。

清空当前步骤的上下文视图。

上下文编排器重新为第一步构建视图，但会保留工作记忆中尚未失效的中间产物（如已生成的代码，前提是用户未否定）。

工作记忆中的每个产物记录依赖条件（如“基于工具列表X和方案Y”）。当用户介入导致依赖条件变化（如用户说“不要用工具A”），相关产物自动标记为无效。

重新进入第一步，但可跳过部分步骤（例如用户只要求修改参数，可复用已有方案直接到第二步修改）。

7. 用户交互与主动介入
7.1 智能体主动发起交互
在任意步骤中，大模型可判断需要用户补充信息、确认风险或选择选项。此时：

暂停当前循环，保存状态（当前步骤的中间产物、上下文视图）。

向用户输出自然语言提问，并等待回复。

用户回复后，将回复作为新消息追加到用户可见历史，恢复状态并从断点继续执行（或重新进入当前步骤）。

7.2 用户主动介入
用户可在任何检查点（步骤之间或第四步执行中）发送新消息，包括：

补充信息、修改目标、取消任务、直接指令、纠错、插入新需求。

处理流程：

中断当前步骤（若为第四步执行中，则终止子进程）。

将新消息追加到用户可见历史。

上下文编排器根据新消息重新评估：是否需要废弃部分工作记忆中的产物（通过依赖追踪）。

重新从第一步开始，但允许快速复用未失效的中间产物（例如用户只改了参数，可跳过第一步，直接进入第二步修改调用计划）。

如果用户要求取消，直接结束循环并输出确认消息。

7.3 交互次数与安全限制
单次任务最多允许 3 次智能体主动提问 + 3 次用户主动介入（可配置），超过则终止并提示用户简化请求。

用户主动取消（“取消”、“停止”）优先处理。

8. 执行沙盒设计
语言：Python 3.10+。

隔离：使用 subprocess 运行临时脚本，设置 timeout、ulimit 限制 CPU 和内存。

危险操作拦截：

代码生成时提示词禁止 import os、subprocess、socket、eval、exec 等，并后过滤。

沙盒环境中，所有工具调用必须通过 tool_gateway，网关内校验参数、权限、频率。

CLI 工具白名单：仅允许执行预注册的命令路径，禁用 shell=True。

网络：沙盒进程默认无网络，需要网络访问的工具（如 HTTP 请求）通过网关代理，可记录审计日志。

输出捕获：捕获 stdout/stderr，限制最大输出大小（如 1MB），超出则截断并警告。

9. 工作记忆与状态管理
9.1 存储结构
工作记忆是一个键值存储，每个条目包含：

id: 唯一标识

type: 工具列表 | 自然语言方案 | 代码 | 执行结果 | 用户偏好

content: 实际数据

step: 产生步骤

dependencies: 依赖的其它条目 id 列表（如代码依赖方案）

timestamp: 创建时间

valid: 是否仍有效（被用户介入标记无效时设为 False）

9.2 状态恢复与回溯
每次检查点保存当前工作记忆的快照（轻量级，仅存储 id 和 valid 状态）。

用户介入后，重新执行第一步时，遍历工作记忆，将 valid=false 的条目及所有依赖它的条目递归标记为无效。

后续步骤只读取 valid=true 的条目，避免使用过时产物。

10. 与 Skill 工具的集成
SKILL.md 被视为一种特殊工具，元数据如下：

L0：技能名称 + 一句话描述（如“撰写周报技能”）

L1：

输入上下文：技能需要哪些信息（如“本周工作项列表”、“上周周报”）

输出：技能生成的内容（如“周报文本”）

内部是否调用其他工具：是/否（若否，则视为纯上下文注入）

L2：预估 token 消耗、是否需用户交互

L3：完整的 SKILL.md 内容（提示词模板、子步骤描述）

调用模式：

模式A（上下文注入）：在第四步执行时，run_skill 函数读取 SKILL.md 作为系统提示词，将输入上下文作为用户消息，调用 LLM 并返回结果。这是默认模式。

模式B（子工作流展开）：若 SKILL.md 包含结构化步骤，可在第三步生成代码时递归展开为对基础工具的调用序列。

渐进式披露：第二步只加载 L1，第三步生成代码时若需详细内容才加载 L3。

11. 完整示例流程
用户：“帮我查一下北京和上海的天气，然后告诉我哪个更暖和，最后把结果发到我的邮箱。”

第一步
加载所有工具 L0，大模型判断需要工具，输出 ["get_weather", "compare_temperature", "send_email"]。

第二步
加载这三个工具的 L1（参数、输出）。

大模型判断为复杂编排（多步、数据依赖），生成自然语言方案：

text
步骤1：调用 get_weather 获取北京天气。
步骤2：调用 get_weather 获取上海天气。
步骤3：调用 compare_temperature 比较两者，得到更暖的城市。
步骤4：调用 send_email 将比较结果发送给用户（收件人从会话历史提取，若无则询问）。
第三步
根据方案和工具 L1，生成 Python 代码（使用 tool_gateway.call），包含错误处理。

第四步
沙盒执行代码。假设获取天气成功，比较成功，发送邮件成功。

第五步
执行结果（成功，邮件已发送） → 大模型生成回答：“已为您比较北京和上海的天气，上海更暖和（25℃ vs 18℃），结果已发送至您的邮箱。”

用户主动介入
若在第二步后用户发送“等一下，不要发邮件，直接显示结果”，则：

新消息追加到用户可见历史。

工作记忆中第二步的方案被标记无效。

重新从第一步开始（但可快速复用工具列表）。

第二步重新生成新方案（去掉 send_email），后续代码不含邮件发送。

最终回答直接显示比较结果。

12. 总结
本设计方案通过五步循环标准化了智能体的决策与执行流程，通过渐进式工具披露和动态上下文编排高效管理信息窗口，通过历史分离保证用户体验与内部推理能力的平衡，通过统一工具模型支持多种工具形式，并通过用户交互与主动介入机制实现人机协作。整体设计具有高灵活性、可扩展性和实用性，适用于复杂任务自动化场景。
